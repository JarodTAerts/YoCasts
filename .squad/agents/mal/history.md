- The repo contains an older Tizen watch app (C#/Xamarin) in `PodcastApp/` with reverse-engineered PocketCasts API work in `PodcastApp/Services/PocketCastsApiService.cs` and `PodcastApp/Accessors/`. The API endpoints and auth flow documented there may be outdated but are a useful starting reference.
- PocketCasts API uses token-based auth (login returns a token used in subsequent requests).
- Known API operations from prior work: login, get subscribed podcasts, get episodes for podcast, get queue (new/unplayed episodes).
- Created comprehensive implementation guide at `docs/garmin-app-implementation-guide.md` covering architecture, constraints, API surface, data flow, UI screens, caching, and build phases.
- Architecture decision: Use `WatchUi.Menu2` for all list views (Queue, Subscribed, Episodes) — it handles scrolling and layout across round/rectangular screens automatically.
- Architecture decision: No text input on watch. User credentials entered via Garmin Connect Mobile app settings (`Application.Properties`), not typed on the watch.
- Architecture decision: Data models are `Dictionary` instances (not classes) — this matches how `makeWebRequest` delivers parsed JSON and avoids unnecessary object overhead.
- Architecture decision: 5-phase build plan — Phase 1 (skeleton + auth), Phase 2 (queue + subscribed), Phase 3 (episode browsing), Phase 4 (polish + hardware test), Phase 5 (enhancements/stretch).
- Architecture decision: Merge PocketCastsApiService and PocketCastsApiAccessor into a single `PocketCastsService.mc` on Garmin — no need for the accessor pattern on a constrained device.
- Architecture decision: Cache strategy uses `Application.Storage` with LRU eviction for per-podcast episode caches. 32 KB per-value limit means trimming episode fields to essentials.
- The Tizen app's `DownloadService.cs` handles episode audio download — this is explicitly deferred to Phase 5 (stretch) on Garmin due to complexity and memory constraints.
- The `Origin: https://playbeta.pocketcasts.com` header is required on PocketCasts API requests (see accessor code).
- **Cross-team update (2026-04-11):** Wash discovered the real queue endpoints are `/up_next/*`, not `/user/new_releases`. Playback sync uses `/sync/update_episode`. This resolves a critical API surface question — the endpoints our service module will use are now fully documented in `docs/pocketcasts-api-reference.md`. Kaylee's UX directly targets these endpoints.
- **Cross-team update (2026-04-12):** Wash completed live API validation (25 endpoints, 20 working). **Critical impacts to architecture:**
  1. **Up Next queue structure is a map, not array** — `/up_next/list` returns `{ order: [...], episodes: {uuid: {...}} }`. Queue view must iterate `order` array and lookup each UUID in `episodes` dictionary. Update PocketCastsService to expose this structure correctly.
  2. **Episode metadata gaps** — `/user/podcast/episodes` returns ONLY status fields (playingStatus, playedUpTo, starred, duration, isDeleted), NO titles/URLs/dates. If podcast view needs titles, must call `POST /user/episode` per-episode or use `podcast-api.pocketcasts.com` bulk endpoint. This impacts episode list cache strategy.
  3. **Token refresh is broken** — `POST /user/token` returns 400. Store credentials on companion phone and re-login if token expires. This affects auth module design.
  4. **Search requires Bearer auth** — `POST /discover/search` returns 401 without token. If discover feature uses search, ensure bearer token is attached.
  5. **Stats values are strings** — `/user/stats/summary` time values are strings, parse to numbers. Minor but affects data coercion in stats module.

- **Token auth & device targeting (2026-04-12):** Wash's deep investigation resolved the `/user/token` 400 error: PocketCasts has two login endpoints — `/user/login` (legacy, no refresh) vs `/user/login_pocket_casts` (OAuth2, full token lifecycle). Refresh via `/user/token` requires `{ "grantType": "refresh_token", "refreshToken": "..." }` body. This impacts PocketCastsService design: store `accessToken`, `refreshToken`, and `expiresIn` in persistent storage; implement proactive refresh before API calls; fallback to re-login on 401. Kaylee also finalized device targeting to Venu 4 41mm (390×390, 768 KB memory). More generous memory budget allows richer token caching strategy in Application.Storage.
- **Offline sync design (2026-04-12):** Created comprehensive offline mode & sync reconciliation architecture at `docs/offline-sync-design.md`. Core design decisions: (1) "Furthest position wins" for playback conflicts — `max(localPos, serverPos)`, no user prompts. (2) Status hierarchy: COMPLETED > IN_PROGRESS > NOT_PLAYED, completed is terminal. (3) Structured changelog in Application.Storage with per-episode coalescing to bound growth. (4) Server authoritative for metadata/subscriptions, watch authoritative for playback state (participates in max-based resolution). (5) Queue reconciliation uses server order as base, removes locally-completed episodes, merges phone-side additions. (6) Push-then-pull sync protocol with 7-step state machine, fully idempotent, safe to retry. (7) Four implementation phases: metadata caching → position tracking + sync → audio download via Media module → full reconciliation. This elevates audio download from Phase 5 stretch goal to Phase 3 in the offline architecture.

- **Wi-Fi direct design update (2026-04-13):** Updated `docs/offline-sync-design.md` to v1.1 for Venu 4 Wi-Fi direct capabilities. Key architectural changes: (1) Three-state connectivity model (Wi-Fi / Phone BT / Fully Offline) replaces binary online/offline. (2) `connectionAvailable` from `System.getDeviceSettings()` is the primary connectivity check — true for either Wi-Fi or BT, so `makeWebRequest()` works on both transports transparently. (3) Phone companion is no longer required for sync or downloads — Wi-Fi direct handles both. (4) Two download paths: system-triggered SyncDelegate (charger + Wi-Fi) and app-initiated downloads (Wi-Fi detected, no charger). (5) Auto-download Up Next episodes on Wi-Fi with battery guards. (6) Sync state machine gains DOWNLOAD? step after PULL_SERVER. (7) Phase 3 simplified — no phone intermediary needed for episode downloads.

- **Audio download implementation plan (2026-04-13):** Created `docs/audio-download-implementation-plan.md` — comprehensive step-by-step blueprint for the full offline sync + audio download system. Key architecture calls: (1) Five build phases (A-E) with explicit dependencies — Changelog/Position Tracking → Sync Engine → Audio Download → Media Playback → Full Reconciliation. (2) Position save at 15s intervals with adaptive battery scaling (60s at <20%, on-pause-only at <10%). (3) Consolidated positions map replaces per-episode keys — single read for sync. (4) `clearCache()` must become selective (no more `clearValues()`) — changelog, auth, download manifest must survive. (5) App type changes to `audio-content-provider` to enable Media module. (6) Bug found: `CachedPodcastService._isConnected()` only checks `phoneConnected`, misses Wi-Fi-direct — must change to `ConnectivityManager.isConnected()`. (7) No download resume in v1 — partial downloads restart from scratch. (8) 64 KB background memory: SyncDelegate gets ~15 KB after runtime overhead — strict allocation rules. (9) 8 new files + 9 modified files. (10) Three open questions remain: exact Media download API, precise Storage limit on Venu 4, and binary audio response handling in makeWebRequest.
